# Static Website Shell Design

**Package:** `fcc-router-consumer-awareness`  
**Date:** 2026-07-09  
**Status:** Approved for implementation

## Goal

Build a public-facing static website shell that explains the FCC router Covered List issue to consumers. The site reads from the existing SQLite database, preserves all editorial caveats, and includes source citations on every regulatory claim.

## Decisions

- **Deployment mode:** Static site.
- **Tech stack:** Vanilla HTML/CSS/JS generated by a Python build script using only the Python standard library.
- **Search:** Precomputed JSON search index loaded and queried client-side.
- **Pages:** eight pages — Home, FAQ, Timeline, Waivers, Approvals, Myths, Sources, Search.
- **Visual style:** Clean, neutral, accessible government/educational style.
- **Tests:** Static-site build + link smoke test, plus JSON structure, HTML escaping, required content, and repeatable-build tests.

## Architecture

```text
SQLite DB
    │
    ▼
scripts/export_site_json.py  ──►  site-data/*.json
    │
    ▼
scripts/build_html.py        ──►  site/*.html + site/static/*
```

- `scripts/export_site_json.py` is the only component that reads the SQLite database.
- `scripts/build_html.py` reads only the generated JSON files. It never touches SQLite.
- `scripts/build_site.py` orchestrates the two steps in order.

## Template strategy

`build_html.py` uses `string.Template` from the Python standard library.

- Each page template is a small HTML fragment with `$key` placeholders.
- `base.html` provides the HTML shell, navigation, footer, and caveat banner.
- A `render()` helper loads a fragment, substitutes placeholders with HTML-escaped JSON values, and inserts the result into `$body` of `base.html`.

All values injected into HTML are escaped with `html.escape` before substitution.

## Page content mapping

| Page | Primary JSON source | Content |
|---|---|---|
| Home | `current_status.json`, `alerts.json`, `faqs.json`, `timeline.json` | Status card, active alerts, first 3 FAQs by `display_order`, latest 3 timeline events by `event_date` descending |
| FAQ | `faqs.json` | FAQs grouped by `category`, with source links |
| Timeline | `timeline.json` | Chronological regulatory events with consumer impact notes |
| Waivers | `waivers.json` | Active/proposed waivers |
| Approvals | `conditional_approvals.json` | Active Conditional Approvals with scope and dates |
| Myths | `claims.json` | Claim/verdict/explanation/consumer guidance table |
| Sources | `sources.json` | Primary-source document library |
| Search | `search_index.json` | Client-side full-text search results |

## Search

1. `export_site_json.py` exports a new `search_index.json` from the FTS5 `search_index` virtual table.
2. Each entry includes `table_name`, `row_id`, `title`, and `snippet`.
3. `search.js` fetches `search_index.json`, performs client-side keyword matching (case-insensitive token containment across title and snippet), and renders grouped results with highlighted snippets.
4. Search requires no server-side component.

## Caveats and source citations

Every page must display:

- "Current as of 2026-07-09"
- "Informational only — not legal advice"
- "Not a SKU-level compliance database; check live FCC records before making model-specific claims"

Every regulatory claim, FAQ answer, timeline event, waiver, and approval must link to its primary source from the `sources` data.


## Testing

### `tests/test_export.py`

- Required JSON files are produced.
- Each JSON file has the expected top-level structure and required keys.
- Datasets are non-empty for the seeded database.
- Export output is deterministic: the same database produces the same JSON bytes.

### `tests/test_build.py`

- HTML escaping: strings containing `<`, `>`, `&`, and quotes are rendered safely.
- Required page content: each generated page contains its expected title, nav link, and key section.
- Internal links: every `href` pointing to a relative path resolves to a generated file or directory.
- Repeatable builds: running `build_html.py` twice on the same JSON produces identical output.

### `tests/test_site.py`

- End-to-end: runs `build_site.py` into a temporary directory.
- Verifies all expected HTML files exist.
- Verifies `site-data/*.json` and `site/static/*` assets are present.

## Risks and constraints

- The site must not be presented as a product/model compliance checker.
- All dates and claims are tied to the seeded `current_as_of` value.
- Client-side search performance depends on the size of `search_index.json`; the current dataset is small enough for in-browser search.

## Out of scope

- Live FCC equipment authorization lookups.
- User accounts, comments, or analytics.
- Backend API changes beyond the existing `sqlite_api.py`.
- JavaScript framework build tooling.
